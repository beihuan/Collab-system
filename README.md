## 1. 系统概述

### 1.1 问题定义

三人开发团队推进两个 MVP 项目，当前痛点：

- **信息孤岛**：三人各自用 Agent 工作，产出分散在各自的 Agent 会话中，没有统一信息汇聚点
- **协调缺失**：缺乏全局视角的角色统筹任务分配、优先级排序和依赖关系管理
- **流程缺失**：没有标准化的进度上报和任务接收流程，每日会议效率低、对齐不彻底
- **能力短板**：技术背景不足、表达能力不足、项目管理能力不足

### 1.2 系统目标

构建一个基于飞书多维表格 + OpenClaw Agent 的协作系统，实现：

1. **Single Source of Truth**：飞书多维表格作为项目进度的唯一真实来源
2. **自动化协调**：PM Agent 作为产品经理兼项目经理，自动规划、分配、跟踪任务
3. **结构化沟通**：通过飞书群组 IM + 飞书文档实现结构化的进度上报和任务指派
4. **人类在环**：所有关键操作需人类确认，PM Agent 通过苏格拉底式追问持续学习项目上下文
5. **深度协作**：开发者 Agent 自动关联代码提交、PR 状态，主动上报阻塞

### 1.3 团队角色

| 角色 | 人员 | 职责 | Agent 运行环境 |
|------|------|------|---------------|
| 产品经理 + AI Coding 开发 | 你（Person A） | 统筹两个项目、全栈开发 | OpenClaw（你的电脑） |
| AI Coding 开发 | Person B | 两个项目的全栈开发 | OpenClaw（Person B 的电脑） |
| 后端开发 | Person C | 后端开发、数据库维护 | OpenClaw（Person C 的电脑） |

### 1.4 技术栈

| 组件 | 技术 | 用途 |
|------|------|------|
| 项目管理 | 飞书多维表格 (Bitable) | 任务进度的 Single Source of Truth |
| 文档协作 | 飞书云文档 | 任务指派、进度报告、会议纪要 |
| 即时通信 | 飞书群组 IM | Agent 间实时通信、通知 |
| CLI 工具 | 飞书 CLI (lark-cli) | 操作多维表格、文档、IM |
| Agent 框架 | OpenClaw | 运行 PM Agent 和开发者 Agent |
| 定时调度 | OpenClaw Cron Job | 定时触发工作流 |
| 周期检查 | OpenClaw Heartbeat | 周期性状态检查 |
| 自定义工作流 | OpenClaw Skill | 封装各 Agent 的专业能力 |

### 2.3 关键设计决策

| 决策项 | 选择 | 理由 |
|--------|------|------|
| PM Agent 运行环境 | 你的电脑上的 OpenClaw | 你是产品经理，PM Agent 与你最近，方便交互确认 |
| Agent 间通信 | 飞书群组 IM + 飞书文档 | 结构化文档保证信息完整性，IM 保证实时通知 |
| 多维表格组织 | 一个应用一个表，项目字段区分 | 便于跨项目查看和 PM 统筹 |
| 认证方式 | 机器人身份 | 操作记录清晰区分人工和 Agent |
| 确认机制 | 分级确认（低风险自动/高风险人工） | 平衡效率和安全 |
| 追问深度 | 深层追问 | 充分理解项目上下文，减少后续偏差 |

## 3. 飞书多维表格设计

### 3.1 表结构

**应用名称**: `项目协作管理`

**数据表名称**: `任务看板`

| 字段名 | 字段类型 | 说明 | 示例值 |
|--------|---------|------|--------|
| task_id | 自动编号 | 唯一任务标识 | TASK-0001 |
| project | 单选 | 所属项目 | 项目A / 项目B |
| title | 文本 | 任务标题 | 实现用户登录API |
| description | 文本 | 详细任务描述 | 基于JWT实现用户登录，支持邮箱/手机号/OAuth... |
| assignee | 单选 | 负责人 | Person A / Person B / Person C |
| status | 单选 | 任务状态 | 待开始 / 进行中 / 待评审 / 已完成 / 已阻塞 |
| priority | 单选 | 优先级 | P0-紧急 / P1-高 / P2-中 / P3-低 |
| dependencies | 文本 | 前置依赖任务ID | TASK-0005, TASK-0008 |
| due_date | 日期 | 截止日期 | 2026-04-25 |
| acceptance_criteria | 文本 | 验收标准 | 1. 接口返回有效JWT\n2. 单测覆盖率>80% |
| milestone | 单选 | 所属里程碑 | MVP-核心功能 / MVP-完善功能 / MVP-上线准备 |
| pr_link | 超链接 | 关联PR | https://github.com/.../pull/42 |
| blocker | 文本 | 阻塞描述 | 第三方OAuth沙箱环境不稳定 |
| estimated_hours | 数字 | 预估工时(小时) | 8 |
| actual_hours | 数字 | 实际工时(小时) | 10 |
| created_at | 创建时间 | 创建时间 | 2026-04-20 |
| updated_at | 修改时间 | 最后更新时间 | 2026-04-20 |
| progress | 单选 | 进度百分比 | 0% / 25% / 50% / 75% / 90% / 100% |
| risk_level | 单选 | 风险等级 | 无风险 / 低风险 / 中风险 / 高风险 |

| 视图名称 | 类型 | 筛选/分组 | 用途 |
|---------|------|----------|------|

| 按项目看板 | 看板视图 | 按project筛选，按status分组 | 单项目进度 |
| 按负责人看板 | 看板视图 | 按assignee分组 | 个人任务视图 |

## 4. PM Agent 设计

### 4.1 PM Agent 概览

PM Agent 运行在你的电脑上的 OpenClaw 实例中，承担产品经理兼项目经理的角色。

**核心职责**:
1. 晨间规划：读取多维表格，生成每日任务指派
2. 进度审核：审核开发者进度报告，更新多维表格
3. 会议辅助：生成会前摘要，处理会后纪要
4. 项目理解：维护项目上下文文档，持续学习
5. 表格管理：多维表格的 CRUD 操作
6. 苏格拉底学习：被纠正时深层追问，信息不完整时主动追问

### 4.2 PM Agent Skill 清单

| Skill 名称 | 触发方式 | 功能 |
|------------|---------|------|
| pm-morning-planner | Cron 08:00 | 晨间任务规划与指派 |
| pm-progress-reviewer | Cron 19:00 | 审核进度报告，更新表格 |
| pm-meeting-assistant | Cron 09:30 + 手动 | 会前摘要生成，会后纪要处理 |
| pm-context-manager | 手动 + Heartbeat | 项目上下文维护 |
| pm-bitable-manager | 被其他Skill调用 | 多维表格CRUD封装 |
| pm-socratic-learner | 被其他Skill调用 + 手动 | 苏格拉底式追问与学习 |

### 4.4 PM Agent Heartbeat 配置

```markdown
# HEARTBEAT.md

## 检查项

### 1. 飞书群组新消息检查 (每30分钟)
- 检查飞书协作群组是否有新的进度报告、阻塞告警、会议纪要
- 如有新进度报告，提醒人类审核
- 如有阻塞告警，提醒人类处理

### 2. 任务逾期检查 (每30分钟)
- 读取多维表格，检查是否有 due_date < 今天 且 status != 已完成 的任务
- 如有逾期任务，在群组中发送逾期提醒
- 询问人类是否需要调整截止日期或重新分配

### 3. 阻塞任务检查 (每30分钟)
- 读取多维表格，检查 status = 已阻塞 的任务
- 如有阻塞超过24小时的任务，提醒人类介入

### 4. 项目上下文完整性检查 (每天1次)
- 检查项目上下文文档是否需要更新
- 如有未明确的决策或信息，向人类追问
```

## 5. 开发者 Agent 设计

### 5.1 开发者 Agent 概览

每个开发者的 OpenClaw 实例中安装相同的 Skill 集，但配置不同的身份信息。

**核心职责**:
1. 进度上报：定时收集工作进度，生成结构化报告
2. 任务接收：读取 PM Agent 发送的任务指派
3. 阻塞通知：检测到阻塞时主动通知 PM Agent
4. 代码活动追踪：关联 git 提交和 PR 状态到任务

### 5.2 开发者 Agent Skill 清单

| Skill 名称 | 触发方式 | 功能 |
|------------|---------|------|
| dev-progress-reporter | Cron 18:00 | 生成并发送进度报告 |
| dev-task-receiver | Cron 09:00 + Heartbeat | 读取并展示任务指派 |
| dev-blocker-notifier | Heartbeat + 手动 | 检测并上报阻塞 |
| dev-code-activity-tracker | Heartbeat | 追踪 git 提交和 PR |

### 5.3 开发者 Agent Cron 配置

```json
{
  "cron": [
    {
      "name": "evening-progress-report",
      "schedule": "0 18 * * 1-5",
      "skill": "dev-progress-reporter",
      "description": "工作日每天18点生成进度报告"
    },
    {
      "name": "morning-task-check",
      "schedule": "0 9 * * 1-5",
      "skill": "dev-task-receiver",
      "description": "工作日每天9点检查今日任务"
    }
  ]

```

### 5.4 开发者 Agent Heartbeat 配置

```markdown
# HEARTBEAT.md

## 检查项

### 1. 新任务指派检查 (每30分钟)
- 检查飞书群组是否有 PM Agent 发来的新任务指派
- 如有，提醒人类查看

### 2. 阻塞检测 (每30分钟)
- 检查当前工作是否有阻塞（如编译错误、依赖缺失、API不通）
- 如检测到阻塞，提醒人类确认后发送阻塞告警

### 3. Git 活动追踪 (每30分钟)
- 扫描本地 git 仓库的最近提交
- 将提交信息与多维表格中的任务ID关联
- 记录到本地工作日志
```

## 8. 苏格拉底式追问机制

### 8.1 触发场景

| 场景 | 触发条件 | 追问深度 |
|------|---------|---------|
| 人类纠正 | 人类说"不对"/"不是这样" | 深层追问 |
| 信息不完整 | 任务描述缺少关键信息 | 深层追问 |
| 进度偏差 | 实际进度与计划偏差>20% | 中层追问 |
| 新任务创建 | 缺少验收标准或依赖关系 | 中层追问 |
| 项目理解更新 | 发现与已有理解矛盾的信息 | 深层追问 |

### 8.2 追问层级定义

**浅层追问** — 确认事实
> "正确的优先级是什么？"

**中层追问** — 理解原因和影响
> "为什么这个优先级更高？它依赖哪些前置任务？调整后对其他任务有什么影响？"

**深层追问** — 挖掘底层逻辑和系统性影响
> "这个优先级调整背后的业务考量是什么？它对整个MVP里程碑有什么影响？我们是否需要重新评估其他任务的优先级？这个决策是否反映了我们对用户需求理解的更新？"

### 8.3 追问交互示例

**场景1: 人类纠正任务优先级**


```bash
# 1. 在各自的OpenClaw skills目录下创建开发者Agent的Skills
cd ~/.openclaw/skills/

# 2. 复制所有开发者Agent Skill文件夹
cp -r dev-progress-reporter/ ./
cp -r dev-task-receiver/ ./
cp -r dev-blocker-notifier/ ./
cp -r dev-code-activity-tracker/ ./

# 3. 配置身份信息
# 编辑 config/identity.json，填入各自的姓名、角色、仓库路径等

# 4. 配置飞书CLI连接信息
# 编辑各Skill中的references/feishu-config.md

# 5. 配置Cron Jobs和Heartbeat
# 按5.3和5.4节配置
```

### 9.4 初始化流程

1. **PM Agent 初始化对话**（一次性）:
   ```
   人类: 我需要初始化PM Agent的项目上下文

   PM Agent: 好的，让我先了解一下项目。我会通过阅读代码库和文档来获取初始信息，然后向你确认。

   [PM Agent 读取代码库和已有文档]

   PM Agent: 我已经阅读了项目代码库和文档，以下是我的初步理解：
   [展示理解]

   我有以下问题需要确认：
   1. ...
   2. ...
   [苏格拉底式追问]

   [多轮对话直到PM Agent充分理解项目]
   ```

2. **飞书多维表格初始化**:
   - 手动创建初始任务记录，或通过飞书CLI批量导入
   - PM Agent 读取并验证表格结构

3. **端到端测试**:
   - 手动触发各Skill，验证完整流程
   - 检查飞书群组消息是否正确发送
   - 检查多维表格是否正确更新
